# structure.ts・質問タイトルルール

本ドキュメントは、`docs/src/structure.ts` の管理ルールと質問タイトルの作成ルールを定義する。
**structure.ts を編集する際は、必ずこのドキュメントを参照すること。**

---

## ダッシュボードの目的

受講生のブラウザのローカルストレージに保存された、**各章の達成目標の進捗率を表示** することが目的である。
研修全体の設計として、**全ての達成目標が達成済みであれば研修達成** とする。

---

## structure.ts の役割

`docs/src/structure.ts` が **達成目標のマスタデータ** である。
各章（Topic）には、複数の達成目標（Question）が紐づく。

---

## structure.ts への追加ルール

**必ず structure.ts に対応する Topic と Questions を追加** すること。

### 追加例（TypeScript形式）

```typescript
withAutoIds({
  id: "variables_and_types",              // ファイル名に対応（例：variables_and_types.mdx）
  label: "変数と型",                       // ダッシュボードに表示されるラベル
  category: "java/basics",                 // カテゴリ（後述）
  questions: [
    {
      title: "変数とは何かを説明できる",  // 達成目標の内容（答えが見えないように注意）
      type: "KNOW",                        // KNOW（知識・理解）/ READ（読解）/ WRITE（実装）
      difficulty: Difficulty.Easy,         // Easy / Medium / Hard
    },
    {
      title: "変数に値を代入できる",
      type: "WRITE",
      difficulty: Difficulty.Easy,
    },
    // ... 他の達成目標
  ],
})
```

## 質問タイトルの作成ルール（最重要）

質問タイトルは、**ダッシュボードで「理解が必要なこと一覧」として活用される** ため、以下の要件を満たす必要がある：

### 要件

1. **機能的要件**：ダッシュボードから演習問題へのリンクとして機能する
2. **教育的要件**：受講生が頭の中で答えを言えるか確認する自己チェックリストとして機能する
3. **UI要件**：タイトルを眺めながら、「これわかるかな？」と自問自答できる

### 核心原則：答えを隠す

質問タイトルに **具体的な答え（型名、キーワード、メタファー）を含めてはならない**。
受講生がタイトルを見ただけでは答えがわからないが、学習後に「ああ、あれか！」と思い出せる形にする。

### ❌ 避けるべきタイトルの例

| 悪い例 | 理由 |
|--------|------|
| 「変数は「データを入れる箱」だとわかる」 | 答えが「箱」だと見えてしまう |
| 「MVCはModel・View・Controllerの略だとわかる」 | 答えが見えてしまう |
| 「&&（かつ）、\|\|（または）を使える」 | 意味が見えてしまう |
| 「カプセル化は「データを隠す」ことだとわかる」 | 答えが見えてしまう |
| 「int型で整数を扱える」 | 答えの型名「int」が見えてしまう |
| 「finalで変更できない定数を作れる」 | 答えのキーワード「final」が見えてしまう |

### ✅ 推奨されるタイトルの例

| 良い例 | 説明 |
|--------|------|
| 「変数とは何かを説明できる」 | 答えを隠している |
| 「MVCの各要素を説明できる」 | 答えを隠している |
| 「&&と\|\|の演算子を使える」 | 意味を隠している |
| 「カプセル化とは何かを説明できる」 | 答えを隠している |
| 「整数型の変数を宣言できる」 | 「int」を隠している |
| 「定数を宣言できる」 | 「final」を隠している |

### ルール詳細

1. **型名・キーワードを隠す**：「int型」「double型」「final」「static」などの具体的な答えを含めない
2. **メタファー表現を避ける**：「箱」「設計図」などの答えになる比喩を使わない
3. **括弧内の説明を避ける**：「（かつ）」「（使える範囲）」のように答えを示す注釈を入れない
4. **「〜とは何かを説明できる」を活用**：知識を問う場合の標準形式
5. **具体的な手段を隠す**：「どうやって〜するか」を答えにしない
6. **20-40文字程度**：簡潔でわかりやすく
7. **行動ベース**：「〜できる」「〜を説明できる」「〜がわかる」の形式
8. **各章の序盤にKNOW型の質問を多めに配置**：自信構築を促進

## Question の 3 つのタイプ

| タイプ | 意味 | 例 |
|--------|------|-----|
| **KNOW** | 知識・理解 | 「変数とは何かを説明できる」「継承とは何かを説明できる」 |
| **READ** | 読解 | 「条件分岐のコードを読んで結果を予測できる」 |
| **WRITE** | 実装 | 「変数に値を代入できる」「if文で条件によって処理を分けられる」 |

### タイプ別の注意点

- **KNOW**: 「〜とは何かを説明できる」の形式を基本とし、答えとなるメタファーや定義を含めない
- **READ**: コードを読んで理解・予測する能力を問う（例：「ループのコードを読んで何回実行されるかわかる」）
- **WRITE**: 実際にコードを書く能力を問う（例：「for文で同じ処理を繰り返せる」）

### 各章の質問順序

- 序盤に KNOW 型の質問を多めに配置し、段階的に WRITE 型に移行する
- 最初の 1-3 問はできるだけ簡単な KNOW 型または WRITE 型にして自信をつけさせる
