# プログラミングブートキャンプ教材 - プロジェクト固有指示

このファイルは、本プロジェクトで教材を作成・編集する際にClaudeCodeが常に参照すべき指示をまとめたものである。

## プロジェクト概要

本プロジェクトは、**プログラミング未経験者（特に文系出身者）**を対象とした、約1〜3ヶ月間のプログラミングブートキャンプ用教材である。
Docusaurusを用いたWebサイト形式で、Java、SQL、HTML/CSS、Git、Spring Frameworkを学習する。

### プロジェクトのコンセプト
- **原則から説明**：プログラミングの大前提（上から下に実行、インデント、ブロック等）から丁寧に教える
- **必要性から理解**：「なぜその技術が必要か」の背景を説明し、納得感を持って学習できるようにする
- **実践重視**：最終的にチーム開発でWebアプリケーションを作成できるレベルを目指す

### 成果物
- Docusaurus形式のWebサイト教材
- 受講生の進捗を可視化するダッシュボード機能
- 各章の達成目標に対応した演習問題

### 教材の閲覧方針
- **印刷は想定していない**
- **ブラウザ閲覧に最適化**して問題ない
- スクロール、インタラクティブなコンポーネント（OneCompilerCodeBlock等）を積極的に活用
- ページの長さは気にせず、教育的に適切な粒度で章を構成

### カリキュラムスケジュール
カリキュラム全体のスケジュール（Day 1-29）については、`.claude/curriculum/schedule.md`を参照すること。

## 対象者の特性

### 受講生プロフィール
- **大半が文系出身のプログラミング未経験者**
- 「変数」「関数」「オブジェクト」などの概念に初めて触れる
- プログラムが「上から下に実行される」といった基本原則も知らない可能性が高い

### 教材作成時に配慮すべきこと
1. **大前提から説明する**
   - 「プログラムは上から下に実行される」
   - 「インデント（字下げ）で構造を表現する」
   - 「`{}`（波括弧）でブロックを作る」
   - など、経験者には自明なことも丁寧に説明する

2. **技術の必要性から理解させる**
   - 「その技術がないと何が困るのか」を先に説明する
   - 例：ループがないと「同じコードを100回書く必要がある」→「だからforループが必要」
   - 例：クラスがないと「データと処理がバラバラで管理が大変」→「だからクラスで整理する」
   
   **説明の具体例:**

   | 技術 | 「ないと何が困るか」 | 「だから必要」 |
   |------|---------------------|---------------|
   | **ループ（for, while）** | 同じ処理を100回書く必要がある | 繰り返しを簡潔に書くためにループが必要 |
   | **メソッド** | 同じコードを何度もコピペする必要がある | 再利用可能にするためにメソッドが必要 |
   | **クラス** | データと処理がバラバラで管理が大変 | データと処理をまとめるためにクラスが必要 |
   | **継承** | 似たようなクラスを何度も書く必要がある | 共通部分を再利用するために継承が必要 |
   | **例外処理** | エラーが起きたときプログラムが止まってしまう | エラーに対処するために例外処理が必要 |

3. **専門用語には必ず注釈を入れる**
   - 例：「コンパイル（人間が書いたコードを機械語に変換すること）」
   - 例：「インスタンス（クラスという設計図から作られた実体）」

## 文体ルール

### 基本方針
- **「である調」で統一**
- 簡潔でわかりやすい文章
- 冗長な表現を避け、本質を端的に伝える

### 良い例と悪い例

#### ✅ 良い例
```
変数とは、データを一時的に保存するための箱である。
プログラムでは、数値や文字列をこの箱に入れて、後で取り出して使う。
```

#### ❌ 悪い例
```
変数というのは、データを一時的に保存するための箱のようなものですね。
プログラムでは、数値や文字列をこの箱に入れて、後で取り出して使うことができるんです。
```

### 専門用語の注釈例
- ❌ 「コンパイルする」
- ✅ 「コンパイル（人間が書いたコードを機械語に変換すること）する」

## ダッシュボードとstructure.tsの管理ルール

### ダッシュボードの目的
受講生のブラウザのローカルストレージに保存された、**各章の達成目標の進捗率を表示**することが目的である。
研修全体の設計として、**全ての達成目標が達成済みであれば研修達成**とする。

### structure.tsの役割
`docs/src/structure.ts`が**達成目標のマスタデータ**である。
各章（Topic）には、複数の達成目標（Question）が紐づく。

### structure.tsへの追加ルール

**必ずstructure.tsに対応するTopicとQuestionsを追加**すること。

#### 追加例（TypeScript形式）
```typescript
withAutoIds({
  id: "variables_and_types",              // ファイル名に対応（例：variables_and_types.mdx）
  label: "変数と型",                       // ダッシュボードに表示されるラベル
  category: "java/basics",                 // カテゴリ（java/basics, java/oop, java/stdlib, db, git, frontend, spring）
  questions: [
    {
      title: "変数とは何かを説明できる",  // 達成目標の内容
      type: "KNOW",                        // KNOW（知識・理解）/ READ（読解）/ WRITE（実装）
      difficulty: Difficulty.Easy,         // Easy / Medium / Hard
    },
    {
      title: "基本データ型（int, double, boolean, String）を理解している",
      type: "KNOW",
      difficulty: Difficulty.Easy,
    },
    {
      title: "変数の宣言と初期化を正しく実装できる",
      type: "WRITE",
      difficulty: Difficulty.Easy,
    },
    // ... 他の達成目標
  ],
})
```

### Questionの3つのタイプ

| タイプ | 意味 | 例 |
|--------|------|-----|
| **KNOW** | 知識・理解 | 「変数とは何かを説明できる」 |
| **READ** | 読解 | 「与えられたコードの実行結果を予測できる」 |
| **WRITE** | 実装 | 「変数を使ったプログラムを実装できる」 |

## 教材作成時の注意事項

### 1. 章立ては教育的に適切な粒度で
- カリキュラムスケジュールは「教えるべき内容」の参考資料
- 章構成は一般的なJava教材として教育的に適切な粒度で設計する
- スケジュールの「Day X」に縛られず、自然な章分けを行う

### 2. 各章の構成
各教材ファイル（`.mdx`）には、以下のセクションを含めること：

```markdown
# 章タイトル

## この章で学ぶこと
- 学習目標を箇条書き

## なぜこの技術が必要か
- 技術の必要性・背景を説明

## 基本概念
- 技術の基本的な説明

## 実例
- サンプルコード（できるだけ実行可能なもの）

## まとめ
- 章の要点を振り返り

## 演習（リンク）
- `docs/src/questions/`配下の演習問題へのリンク
```

### 3. サンプルコードの方針
- 可能な限り**実行可能なコード**を提示する
- OneCompilerコンポーネントを活用し、ブラウザ上で実行できる形にする
- コードには適切なコメントを付ける

### 4. MDXファイルの記述ルール

#### import文
- OneCompilerCodeBlockのimport文は以下の形式で記述する：
  ```typescript
  import { OneCompilerCodeBlock } from "@site/src/components/OneCompilerCodeBlock";
  ```
- import文は必ずファイルの先頭に配置する

#### 改行の扱い
- **MDXファイルで改行を反映させるには、改行の前に半角スペースを2つ入れる**
- これはMarkdownの仕様で、半角スペース2つ（`  `）+ 改行で`<br>`タグが生成される

**例：**
```markdown
この章では、プログラミングとは何か、Javaとは何かを学び、  
最初のプログラム「Hello World」を書いて実行する。
```

上記のように、1行目の末尾に半角スペース2つを入れることで、2行目が改行として表示される。

**適用箇所：**
- 複数行にわたる説明文
- 箇条書きの各項目内で複数行になる場合
- 表内で複数行になる場合

**注意：**
- 段落を分ける場合（空行を入れる場合）は、半角スペース2つは不要
- コードブロック内では不要

#### 強調記号の扱い

MDXファイルで **`**`（強調記号）** を使用する際、プレビューで正しく展開されない場合がある。

**問題が起きるケース：**
- 句読点（`。`）、閉じ括弧（`」`、`）`）などの直後に`**`を配置した場合
- 例：`技術である**必要性から理解**させる`（展開されない可能性）

**解決策：**
- **`**`の前後に半角スペースを置く**

**正しい例：**
```markdown
技術である **必要性から理解** させる
この章では、 **プログラムの基本原則とJavaの第一歩** を踏み出そう。
「これは **重要** である」
```

**適用ルール：**
- 強調記号`**`の前後には必ず半角スペースを1つ入れる
- 文頭や文末の場合は、前後どちらか一方のみで良い
- 例外：バッククォート`` ` ``と組み合わせる場合（例：`` **`main`メソッド** ``）は、バッククォートとの間にスペースは不要

#### 表の中での特殊文字

MDXファイルの表（table）の中で特殊文字を使用する際は、エスケープに注意が必要である。

**問題が起きるケース：**
- 表の列区切り文字`|`を表示したい場合
- 例：`**\|\|**`（論理OR演算子）をエスケープして表示しようとする

**エラー例：**
```markdown
| 演算子 | 意味 |
|--------|------|
| **\|\|** | OR演算子 |  ← エラー！MDXパーサーが混乱する
```

**解決策：**
- **バッククォート（コード記法）を使う**
- HTMLエンティティ（`&#124;`）は避け、可能な限りバッククォートで囲む

**正しい例：**
```markdown
| 演算子 | 意味 |
|--------|------|
| `||` | OR演算子 |
| `&&` | AND演算子 |
| `!` | NOT演算子 |
```

**適用ルール：**
- 表の中でプログラミングの演算子や記号を表示する場合は、バッククォートで囲む
- `\|`のようなエスケープは使わない
- 一貫性のため、同じ列の他の要素もバッククォートで統一することを推奨

#### 演算子や特殊記号の記述

MDXファイルで演算子や特殊記号を記述する際は、適切なエスケープが必要である。

**問題が起きるケース：**
- `<`や`>`がHTMLタグとして解釈される
- 例：`比較演算子（==, !=, <, >, <=, >=）`（MDXパーサーがエラー）

**解決策（2つの方法）：**

1. **バッククォートで囲む**（推奨）
   ```markdown
   - 算術演算子（`+`, `-`, `*`, `/`, `%`）を使いこなせる
   - 比較演算子（`==`, `!=`, `<`, `>`, `<=`, `>=`）を理解している
   - 論理演算子（`&&`, `||`, `!`）を理解している
   ```

2. **バックスラッシュでエスケープする**
   ```markdown
   - 比較演算子（==, !=, \<, \>, \<=, \>=）を理解している
   ```
   - HTMLで特別な意味を持つ文字（`<`、`>`、`&`など）は`\`でエスケープできる

**適用ルール：**
- **基本的にはバッククォートで囲む方法を推奨**（可読性が高く、一貫性を保ちやすい）
- バッククォートを使う場合は、同じ文脈内のすべての演算子を統一的に囲む
- 文章中で演算子を説明する際は、バックスラッシュエスケープでも可
- 特に`<`、`>`、`&`などHTMLで特別な意味を持つ文字は必ず何らかの方法でエスケープする

### 6. Docusaurus admonitionsの活用

#### admonitionsとは

Docusaurusには、特定の情報を強調表示するための **admonitions**（注意書き）機能がある。
これを積極的に活用することで、受講生にとって重要な情報を視覚的に目立たせることができる。

#### 利用可能なadmonitionsの種類

| admonition | 用途 | 色 |
|------------|------|-----|
| `:::note` | 補足説明、参考情報 | 青 |
| `:::tip` | ヒント、推奨事項 | 緑 |
| `:::info` | 一般的な情報、共通概念 | 青 |
| `:::caution` | 注意が必要な事項 | 黄 |
| `:::warning` | 警告、よくある間違い | 赤 |

#### 使い方

```markdown
:::note 補足
用語の補足説明や、技術的な背景情報を記載する。
:::

:::tip ポイント
学習のコツや、効率的な方法を示す。
:::

:::info 実用的な情報
プログラミングの本質的な理解を深める情報や、実務での活用方法を記載する。
:::

:::caution 重要
特に注意すべき仕様や、間違えやすいポイントを強調する。
:::

:::warning 注意
よくある間違いや、エラーの原因となる事項を警告する。
:::
```

#### 活用方針

1. **`:::info`**：実用的で本質的な情報を提供
   - **使用目的**：初学者が技術の本質を理解するための実用的な情報を記載
   - **記載内容の例**：
     - 技術の設計思想や背景（例：「なぜオブジェクト指向が必要か」）
     - 内部実装の理解（例：「ArrayListは内部で配列を使う」）
     - 実務での活用方法（例：「インターフェースはDIやテストで活用される」）
     - パフォーマンスや保守性の注意点（例：「オートボクシングは頻繁に発生するとパフォーマンスに影響」）
     - ベストプラクティス（例：「必須のフィールドは引数ありコンストラクタで初期化」）
   - **避けるべき内容**：
     - ❌ 「この概念はほとんどのプログラミング言語で共通である」のような、学習価値の低い一般論
     - ❌ 初学者にとって意味がない情報
   - **目的**：受講生が単なる文法知識ではなく、技術の本質や実用性を理解できるようにする

2. **`:::tip ポイント`**：学習のコツや重要なポイントを強調
   - 例：「最初は用語を全て覚える必要はない。この章を読み進めながら、何度もこの表を見返して、徐々に慣れていこう」
   - 実践的なアドバイスを提供

3. **`:::caution 重要`**：特に注意すべき仕様や動作
   - 例：「インデックスは0から始まり、配列の長さ-1まで」
   - 間違えやすいポイントを事前に注意喚起

4. **`:::warning 注意`**：よくあるエラーや間違い
   - 例：「存在しないインデックスにアクセスすると、ArrayIndexOutOfBoundsExceptionが発生する」
   - エラーを未然に防ぐ

5. **`:::note 補足`**：技術的な背景や追加情報
   - 例：「オブジェクトとインスタンスは、ほぼ同じ意味で使われる」
   - 深い理解を促進

#### 適用ルール

- **適度に使用する**：多用しすぎると、重要度が薄れる
- **1つの章に3〜5個程度**を目安とする
- **特に初学者が間違えやすいポイント**には積極的に使用する
- **`:::info`は学習価値の高い実用的な情報に限定**し、「共通概念」のような一般論は避ける

### 7. Serenaメモリの活用

#### Serenaとは
Serenaは、プロジェクトに関する重要な情報をメモリファイルとして保存・参照できるシステムである。
教材作成の一貫性を保つため、積極的に活用すること。

#### メモリに保存すべき情報
- **完成した章の構成や特徴**（例：Java第1章の教材構成、MDXルール適用状態）
- **設計上の重要な決定事項**（例：なぜこの順序で説明するか、なぜこの例を使うか）
- **頻繁に参照する教材テンプレート**（例：標準的な章構成パターン）
- **プロジェクト固有の用語や表記ルール**

#### メモリの活用方法
1. **新しい章を作成する前に**、関連する既存のメモリを参照する
   - 例：Java第2章を作成する際は、`java_01_java_basics_content`メモリを参照して文体や構成を揃える
2. **重要な章を完成させた後**、その構成や特徴をメモリに保存する
   - `mcp__serena__write_memory`ツールを使用
3. **一貫性が求められる作業**では、必ず過去のメモリを確認する

#### メモリの命名規則
- **章別メモリ**：`java_XX_chapter_name_content`（例：`java_01_java_basics_content`）
- **設計判断メモリ**：`design_decisions_YYYYMMDD`
- **テンプレートメモリ**：`template_chapter_structure`
